# autoschedule Agent 工作规约

本文件是本仓库的最高工作约束。产品事实看 `spec.md`；只有 owner 明确要求计划时才维护 `plan.md`。

## 基础约束

- 始终使用简体中文和项目 owner 交流，除非 owner 明确要求其他语言。
- 阅读文本时使用正确编码；显示异常时先验证编码，不误判文件损坏。
- 不回滚 owner 已有改动，只处理当前任务相关文件。
- commit、push、发布和部署只在 owner 明确授权时执行。
- 禁止机械替换、无审查批量替换和大小写不受控的命名迁移。
- 禁止旧别名、兼容转发、旧语义包装、历史适配层和过渡文件。
- 当前实现只服务当前生效事实。

## 不可突破核心规则
- AGENTS.md 是项目AI协作永久规范，spec.md 是产品需求边界。
- 【严格禁止】在没有获得用户明确书面许可的情况下，自行修改、删减、弱化、曲解 AGENTS.md 和 spec.md 内任何条款。
- 【严格禁止】实现功能时，无视文档里已定义的约束；任何代码实现必须同时满足两份文档全部要求。
- 如果新需求和现有 AGENTS.md / spec.md 存在冲突：
  - 禁止自行决定“以新需求覆盖旧规则”
  - 第一时间列出冲突条目，主动向用户上报冲突，等待用户二选一：
    - 方案A：调整新需求，兼容原有规则
    - 方案B：授权你正式修订 AGENTS.md / spec.md，更新文档后再开发
- 每次开始处理任务前，主动自检：对照 AGENTS.md + spec.md 核对，确认当前方案没有违反历史约束。

## 需求确认机制

- 对功能开发、规则调整、页面交互、数据结构、导入导出或会改变现有行为的需求，禁止仅凭 agent 自行推断后直接修改。
- 在任何写入、实现或配置变更前，必须先向 owner 简要确认：
  - 对需求的具体理解，必要时用一个贴近业务的例子说明预期结果；
  - 本次会修改的范围，以及明确不会改变的既有行为；
  - 关键业务边界、异常场景和可验证的完成效果。
- 需求存在多种合理实现、信息不足、术语可能有歧义，或可能影响已有规则时，必须主动提出针对性问题，等待 owner 明确选择；不得选择“看起来最合理”的方案自行执行。
- 只有在 owner 明确确认上述理解或明确授权“直接执行”后，才可开始修改文件、配置或数据。确认只授权已经说明的范围；后续新增范围、改变既有规则或发现冲突时，必须重新确认。
- 纯粹的解释、只读排查、状态查询和 owner 已明确要求“直接执行”的无歧义小改动，可不额外重复确认；但仍须遵守 AGENTS.md、spec.md 和授权边界。


## 事实纪律

- 判断必须基于 owner 明确输入、当前仓库、命令输出或真实运行结果。
- 区分已确认事实、owner 提供但未验证的事实、推测、计划事项和已完成并验证事项。
- 任何未实现或未验证内容，在任何载体中都不能写成已经实现或已经通过。
- 新事实推翻计划时，先更新 `plan.md`，再继续。

## 文件职责

- `AGENTS.md`：工作规则和协作纪律。
- `spec.md`：当前产品事实、边界和验收标准。
- `plan.md`：当前中大型任务的单文件执行合同。
- `README.md`：面向用户和新读者的使用入口。

## 代码职责

- `src/domain/` 只处理排班、轮值、时间、疲劳和反馈等业务规则，不读取 DOM、不写浏览器存储。
- `src/infrastructure/` 负责浏览器存储、Excel 和分享文件等外部边界，不承载页面事件编排。
- `src/ui/` 只负责页面 HTML 渲染和展示数据，不直接修改应用状态。
- `src/app/configuration-actions.ts` 负责人员、航班、模板、岗位规则及配置字段的状态变更。
- `src/app/schedule-actions.ts` 负责排班结果编辑、人员交换、引导复用和临时岗位状态变更。
- `src/app/history-actions.ts` 负责排班历史记录构造、替换、清空和删除。
- `src/app/workbook-actions.ts` 负责把已解析的工作簿数据应用到应用状态。
- `src/app.ts` 只负责应用生命周期、页面切换、DOM 事件分发、确认/提示和调用上述动作；新增业务规则不得直接堆回该文件。

## Skills

- 项目级 skill 放在 `.agents/skills/`。
- 任务命中 skill 的 `description` 时，必须先读取对应 `SKILL.md` 再行动。
- owner 明确要求计划时使用 `.agents/skills/plan/SKILL.md`，并维护根目录 `plan.md`。
- 修改实现、测试、事实文档或运行配置时使用 `.agents/skills/autoschedule-dev/SKILL.md`。
- 业务专属 skill 只在真实业务需要时新增，不把临时任务或产品事实硬编码进通用 skill。

## 计划规则

- owner 明确要求计划时才创建或维护 `plan.md`；owner 明确免除时不得擅自创建。
- 计划按需求、事实、失败测试、目标、不做范围、设计、任务、验证、收口组织。
- checklist 随进度更新，不用口头说明代替计划更新。

## 工程规则

- 优先沿用仓库已经成立的技术栈、命名和模块边界。
- 单一职责按变化原因判断；文件超过 300 行时审查职责，但不机械拆分。
- 状态、规则、数据、展示、外部接线和错误处理变化原因不同时，拆清边界。
- 测试保护真实规则、边界和核心路径，不保护废弃行为。
- 文档、实现、测试和配置在同一次交付中保持一致。

## 完整验证

- `npm.cmd run typecheck`
- `npm.cmd test`
- `npm.cmd run build`
- `npm.cmd run verify`

收尾前运行项目完整验证和核心路径检查。无法运行时说明具体命令、原因和剩余风险。

## 推送门槛

- 任何实现、测试、文档或运行配置改动，必须先完成上述完整验证，所有命令退出码为 0 后才允许 commit 和 push。
- 任一验证失败、未运行或存在未说明的剩余风险时，不得推送到远端。
- push 只在 owner 明确授权后执行；授权不替代完整验证门槛。

## 安全与交付

- 不泄露密钥、令牌、个人数据或私有配置。
- 破坏性操作、生产变更、权限扩大和付费操作需要明确授权。
- 交付必须说明完成事实、实际验证、未验证项和剩余风险，不交半成品。
